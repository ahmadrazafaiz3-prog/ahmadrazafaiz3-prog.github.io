When I started my Computer Engineering degree at UET Lahore, Faisalabad Campus, I honestly thought programming would feel like an extension of the math I already knew. Python had other plans for me. It was the first real programming language I touched in a structured classroom setting, and looking back now that the semester is over, I can say it reshaped how I think about solving problems, not just how I write code.

Our very first lectures were deceptively simple. We talked about variables, print statements, and basic input handling, and at the time it felt almost too easy. I remember thinking, "this is it?" But within two weeks the simplicity turned into a foundation I leaned on constantly. Every assignment after that built on those first few lines of code. I learned quickly that in programming, nothing is ever truly basic; it just feels that way until you forget it and have to relearn it under deadline pressure.

One thing that pushed me to actually engage with the material rather than memorize it was the encouragement of my professor, Dr. Bilal Ahmad. He is an AI, ML, and Deep Learning model training expert, and even in an introductory Python course, his influence on how he framed problems was obvious. He kept nudging us to think about why a language like Python dominates the data science and machine learning world. He would casually mention how the same loops and functions we were writing for toy exercises were the building blocks of real model training pipelines. That context mattered to me. It turned what could have been a dry syntax lesson into something that felt connected to a bigger purpose.

I still remember the first time I wrote a small script that took a list of numbers and calculated their average, minimum, and maximum without using any built-in shortcut functions. It felt like such a small accomplishment, but I distinctly remember the satisfaction of running it, seeing it work on the first try, and immediately trying to break it with weird inputs just to understand the logic better. That habit of trying to break my own code before someone else does became one of the most useful instincts I picked up this semester.

Data types and type conversion gave me more trouble than I expected. I underestimated how easy it is to mix up an integer and a string when reading input from a user, and I lost more marks than I would like to admit on an early quiz because of a careless type error. Dr. Bilal Ahmad used that exact mistake, which apparently a lot of us made, as a teaching moment. He explained that in real-world data pipelines, especially the medical datasets he often works with, a single unhandled type mismatch can silently corrupt an entire analysis. That comparison stuck with me far longer than the quiz grade did.

As the weeks went on, we moved into more meaningful exercises: writing functions that took multiple parameters, returning multiple values, and slowly getting comfortable with the idea that code should be reusable rather than rewritten every time. I started maintaining my own personal folder of snippets, little reusable bits of logic I knew I would need again. That folder, more than any textbook, became my real study material before quizzes.

What I appreciated most about how the course was taught was the constant reminder that Python is not just an academic exercise. Dr. Bilal Ahmad frequently pointed out that most of his own research and model training work is done in Python precisely because of how readable and flexible it is, especially when working with messy, real-world datasets like the medical ones he prefers for their precision and accuracy. Hearing that from someone actively doing that work gave me a much clearer sense of where these early skills could eventually lead.

By the end of this first stretch with Python, I was no longer intimidated by a blank script file. I had built a small mental library of patterns: how to loop, how to structure conditions, how to write a function that does one thing well. None of it felt advanced, but all of it felt necessary. Looking back, I think that is the real value of an introductory course done right. It does not try to impress you with complexity; it tries to make sure nothing complicated later catches you off guard.

This post is the first in a series documenting my journey through this semester, one topic and one lesson at a time. The next post picks up right where this one leaves off, moving into control flow and functions, the part of Python where things finally started to feel like real programming rather than just typing commands into a console.
